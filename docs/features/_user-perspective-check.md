# ユーザー目線チェック設計書

> 全40機能の定義ファイルに追加する「ユーザー目線チェック」の基準

## 前提: ユーザーは誰か

サイトを訪れるお客さん。テクノロジーのことは何も知らない。
- 工務店を探してるお母さん
- 美容院を予約したいOL
- 飲食店のメニューを見てる会社員
- 歯医者を比較してる高齢者

**彼らはコンポーネントもFramer Motionも見ていない。**
見ているのは「自分の問題が解決しそうか」「この会社を信用していいか」「次に何をすればいいか」だけ。

## 各機能に追加するセクション

### 「## この機能の核」（1文）

この機能が成功したと言える状態を1文で定義する。
技術的な「動く」ではなく、**ユーザーの行動や感情の変化**で書く。

例:
- ヒーロー: 「3秒以内に "ここは自分に関係ある" と感じて、スクロールする」
- FAQ: 「"いくら？" の答えが具体的な数字で見つかり、不安が消える」
- お問い合わせ: 「"ちょっと聞くだけ" のつもりで送信ボタンを押せる」
- 予約: 「30秒で予約が完了し、電話しなくて済む」
- お客様の声: 「"この会社に頼んで大丈夫" と確信する」
- AIチャットボット: 「夜11時でも自分の質問の答えが得られる」
- ニュース: 「日付を見て "この会社はちゃんと動いてる" と安心する」
- CTA: 「"聞いてみよう" と思った瞬間にボタンが目の前にある」

### 「## ユーザー行動チェック」（5〜8項目）

技術チェック（動くか）とは別に、**ユーザーの行動シナリオ**で検証する。

各項目は以下のフォーマット:
```
- 【状況】→【ユーザーの行動】→【期待される結果】→【失敗パターン】
```

例（お問い合わせフォーム）:
```
1. 【初見で見た瞬間】→ フォームの項目数を数える → 5個以下なら「簡単そう」と感じる → 10個以上だと「面倒くさい」で離脱
2. 【入力中に迷う】→ 何を書けばいいかわからず手が止まる → placeholder/ヘルプテキストで書くべき内容がわかる → 真っ白な入力欄だけで途方に暮れる
3. 【間違えて入力した】→ エラーが出る → どこが間違いか・どう直すか即座にわかる → 「入力エラーがあります」としか表示されず、どこかわからない
4. 【送信した後】→ ボタンを押す → 「送信しました。3営業日以内にご連絡します」と表示される → 画面が変わらず、送れたのかわからない
5. 【個人情報が心配】→ 電話番号や住所を入力する前に不安になる → プライバシーポリシーリンクが目に入る → 何の説明もなく個人情報を求められる
```

例（AIチャットボット）:
```
1. 【夜11時にスマホで見てる】→ 質問を打つ → 答えが返る → 「営業時間内にお電話ください」で終わる
2. 【質問の仕方がわからない】→ 何を聞けばいいか迷う → クイックリプライで選択肢が出る → テキスト入力だけで何も手がかりがない
3. 【答えが的外れ】→ 期待した答えと違う → 「この質問はお問い合わせフォームから」と誘導される → ループして同じ答えが返り続ける
4. 【チャットを閉じて再度開く】→ さっきの会話が残ってる → 続きから話せる → 初回挨拶からやり直し
```

### 「## スコアリング改定」（3段階）

スコアリングの5軸はそのまま使うが、判定基準を**ユーザー目線の3段階**に変更:

| レンジ | レベル | 基準 |
|---|---|---|
| **90〜100点** | 価値がある | ユーザーが**目的を達成**できる。行動が完結する。不安が消える。次のアクションに進める。 |
| **80〜89点** | 使える | 機能として**正しく動く**。操作に迷わない。レイアウトが崩れない。だが「おっ」とは思わない。 |
| **70〜79点** | 動く | コードにエラーがない。表示される。だが**ユーザーが迷う場面がある**。離脱ポイントが存在する。 |
| **0〜69点** | 壊れてる | 動かない、見切れる、意味がわからない。 |

**80点未満は作り直し。80〜89点は改善して通す。90点以上でパス。**

### 各5軸の採点を「ユーザー目線」に変更

| 軸 | 配点 | 旧基準 | 新基準 |
|---|---|---|---|
| **機能性** | 30点 | 必須要件を満たしているか | **ユーザーの行動が完結するか。** フォーム送信、予約完了、FAQ回答取得が最後まで辿り着けるか |
| **見た目** | 25点 | プロっぽいか | **信用できそうか。** 初見3秒で「ちゃんとした会社だ」と思えるか。安っぽさ・素人感がないか |
| **レスポンシブ** | 20点 | 崩れないか | **スマホで片手で使い切れるか。** 指で操作するもの（ボタン、スライダー、カレンダー）が44px以上か。横スクロールが発生しないか |
| **コード品質** | 15点 | 型安全か | **遅くないか、壊れにくいか。** パフォーマンスに影響する問題がないか。将来の変更で壊れやすい構造になっていないか |
| **アクセシビリティ** | 10点 | alt属性があるか | **誰でも使えるか。** 目が悪い人がスクリーンリーダーで内容を理解できるか。キーボードだけで操作できるか |
