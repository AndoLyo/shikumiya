# レスポンシブ対応チェック
> ID: responsive-check | カテゴリ: validation | プラン: all（全実装に適用）

## 概要

顧客サイトが全デバイスサイズで正しく表示・操作できることを検証するバリデーション機能。
現在のウェブトラフィックの60%以上がモバイルデバイスからのアクセスであり、
業種を問わずモバイルファーストの対応は必須。

「スマホで見たら崩れていた」は顧客からの信頼を即座に失う致命的な品質問題。
3つの主要テストサイズ（モバイル375px / タブレット768px / デスクトップ1280px）で
全セクションの表示・操作を確認する。
Tailwind CSSのブレークポイント体系（sm:640px, md:768px, lg:1024px, xl:1280px）に
準拠した実装であることを合わせて検証する。

## この機能の核
スマホでもPCでも同じように使える

## チェック項目（18項目）

### 1. 横スクロール（水平オーバーフロー）の不在
- 全ページで横スクロールが発生していない
- 375px / 768px / 1280px の3サイズで確認
- 一部でも横スクロールが出る場合は不合格
- `overflow-x: hidden` でごまかすのではなく、原因を修正

### 2. タッチターゲットサイズ
- ボタン・リンク・フォーム入力欄のサイズが **44px x 44px以上**
- WCAG 2.5.5準拠
- padding込みの最終表示サイズで判定
- テキストリンク・アイコンボタンは特に注意

### 3. モバイルフォントサイズ
- モバイル（375px）でのフォントサイズが **14px以上**
- `text-sm` (14px) が下限
- `text-xs` (12px) は本文テキストに使用不可（注釈・補足のみ許容）
- 行間（line-height）も十分に確保（1.5以上推奨）

### 4. 画像のコンテナ収容
- 画像がコンテナからはみ出していない
- `max-w-full` or `w-full` + `object-cover` の適用
- `next/image` の `fill` 使用時は親に `relative` + `overflow-hidden`
- SVGアイコンも固定幅がコンテナを超えないこと

### 5. ナビゲーションの全サイズ対応
- モバイル: ハンバーガーメニュー
- デスクトップ: 横並びメニュー
- 切り替えのブレークポイントが明確（`md:hidden` / `md:flex` 等）
- タブレットサイズでどちらが表示されるか意図的に決定されている

### 6. ハンバーガーメニューの動作
- 開閉が正しく動作する
- メニュー展開時に背景がスクロールしない（`body` の `overflow: hidden` 制御）
- メニュー外クリック（オーバーレイ）で閉じられる
- メニュー展開中のアニメーションが自然

### 7. フォームのモバイル対応
- input幅が100%（コンテナいっぱい）
- 適切な `type` 属性: `tel` / `email` / `url` / `number`
- 十分な行間と余白（フィールド間 16px以上）
- select要素がモバイルのネイティブピッカーで操作可能
- 送信ボタンがファーストビューまたはスクロール範囲内

### 8. モーダル・ダイアログの表示
- 全画面サイズに収まる
- 閉じるボタンが画面内の到達可能な位置にある
- モバイルで画面より大きくなり×ボタンが見えなくなるケースを防止
- 縦長コンテンツのモーダルは内部スクロール対応

### 9. テーブル要素のモバイル対応
- 横スクロール可能なラッパー（`overflow-x-auto`）を適用
- またはカード形式にリフロー（テーブルを解体して縦積み表示）
- 料金表・比較表・スペック表など情報テーブル全てが対象
- スクロール可能であることのヒント表示（シャドウ等）

### 10. 固定要素の影響
- `position: fixed` / `sticky` の要素がコンテンツを隠していない
- 固定ヘッダー: `padding-top` でコンテンツ開始位置を確保
- 固定CTAバー: `padding-bottom` でコンテンツ最下部が隠れない
- 複数の固定要素が重なってコンテンツ領域を圧迫していない

### 11. 余白のスケーリング
- 余白・パディングがブレークポイントに応じて適切にスケール
- デスクトップの `px-16` がモバイルで `px-4` になる等
- モバイルで過大な余白がない（画面の30%以上が余白は多すぎ）
- セクション間のマージンもスケール（`py-20 md:py-32` 等）

### 12. グリッドレイアウトのカラム変化
- カラム数がブレークポイントで適切に変化
- 例: 実績ギャラリー — モバイル1列、タブレット2列、デスクトップ3〜4列
- カードが潰れて中身が読めなくなるケースを防止
- `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` の明示的な指定

### 13. ヒーローセクション
- テキストがモバイルで切れていない
- CTAボタンがビューポート内に収まっている
- 背景画像のフォーカスポイントがモバイルでも適切
- テキストオーバーフロー・ボタンはみ出しの防止

### 14. フッターのレイアウト変化
- 多カラムレイアウトがモバイルで縦積みに変わっている
- 横並びフッターリンクがモバイルで折り返されて潰れていない
- コピーライト・ロゴ等の表示位置が適切
- SNSアイコンリンクのタッチターゲットが十分

### 15. ギャラリー・スライダーのタッチ操作
- モバイルでスワイプ操作が可能
- タップで拡大表示（ライトボックス）が動作
- ドットインジケーターやページネーションがモバイルで操作可能
- ピンチズームが必要な場合は対応されている

### 16. Tailwindブレークポイントの使用
- `sm:` / `md:` / `lg:` / `xl:` プレフィックスが適切に使用
- マジックナンバーの `@media (min-width: 847px)` のようなカスタムメディアクエリを書いていない
- 一貫したブレークポイント使用（`md:` と `lg:` の使い分けが明確）
- カスタムブレークポイントが必要な場合は `tailwind.config` で定義

### 17. viewportメタタグ
- `width=device-width, initial-scale=1` が設定
- Next.js App Routerの `viewport` export で管理
- `maximum-scale=1` は使用しない（ユーザーのズームを妨げる）
- `user-scalable=no` は使用しない

### 18. フォント・見出しサイズのスケーリング
- ブレークポイントごとにサイズが変化
- 例: `text-2xl md:text-4xl xl:text-5xl`
- 固定サイズの見出しが全デバイスで使われていない
- `clamp()` を使う場合もTailwindの体系内で管理

## 3層チェック

> この機能の核: **スマホでもPCでも同じように使える**
> 核が実現されているかどうかを、3層で検証する。
> ※ バリデーション機能のため、Layer 1 = チェック項目が正しく検証可能か、Layer 2 = チェック結果がUXに反映されるか、Layer 3 = スマホユーザーがPC版と同じことが全部できるか

### Layer 1: チェック項目の検証可能性（正しくチェックできるか）— 30点

全18チェック項目が技術的に検証可能かの確認。全項目PASSが前提条件。1つでもFAILなら全体70点未満（作り直し）。

| # | チェック項目 | 合格基準 | FAIL例 |
|---|---|---|---|
| F-1 | 3ブレイクポイントでの横スクロール検出 | 375/768/1280pxで水平オーバーフローをコード/目視で検出可能 | overflow-x:hiddenで隠してて検出できない |
| F-2 | タッチターゲットサイズの計測 | padding込みの最終表示サイズで44px判定が可能 | CSSの計算値が複雑で実測不能 |
| F-3 | レスポンシブプレフィックスの解析 | sm:/md:/lg:/xl:の使用率をコードから計測可能 | CSS-in-JSで静的解析不能 |
| F-4 | ハンバーガーメニューの存在確認 | md:hidden/md:flexの切替パターンを検出可能 | カスタムJSでの表示切替が検出困難 |
| F-5 | グリッドカラム変化の検出 | grid-cols-*のレスポンシブ定義を解析可能 | 動的classNameで静的解析不能 |
| F-6 | input type属性の確認 | tel/email/url等のtype属性を自動検出可能 | フォームが動的生成で検出困難 |
| F-7 | viewport設定の確認 | layout.tsxのviewport exportを解析可能 | generateViewport使用で静的解析不能 |
| F-8 | 固定要素のpadding対応確認 | fixed/sticky要素と対応padding/marginのペアを検出 | 複数コンポーネントに分散して追跡困難 |

### Layer 2: UXへの反映（モバイルでの操作品質）— 40点

チェック項目をPASSした結果、実際のモバイル操作体験が向上するかの検証。

| # | チェック項目 | 合格基準 | 閾値 | 配点 |
|---|---|---|---|---|
| U-1 | モバイルでの操作性 | タップ・スワイプが快適、誤タップなし | タッチターゲット44px+、隣接要素8px+間隔 | 10点 |
| U-2 | モバイルでの文字可読性 | 全テキストがピンチズームなしで読める | 本文14px以上、行間1.5以上 | 10点 |
| U-3 | PCでのレイアウト品質 | 広い画面を活かした情報配置 | 3〜4列グリッド、適切な余白 | 8点 |
| U-4 | フォームのモバイル入力体験 | 適切なキーボードが表示される | tel→テンキー、email→@キー付き | 6点 |
| U-5 | ナビゲーション操作の自然さ | メニュー開閉がスムーズ | アニメーション+背景ロック+オーバーレイ閉じ | 4点 |
| U-6 | テーブル/料金表のモバイル閲覧 | 全情報にアクセスできる | 横スクロール+ヒント or カードリフロー | 2点 |

### Layer 3: 核の目的達成（スマホで全機能が使えるか）— 30点

この機能の核「スマホでもPCでも同じように使える」が実現されているかの検証。ここが80点と90点を分ける。

| # | チェック項目 | 合格基準 | 90点の例 | 80点の例 | FAIL例 |
|---|---|---|---|---|---|
| V-1 | スマホで全操作が完結 | PC版でできることがスマホでも全てできる | ナビ/フォーム/ギャラリー/CTA全対応 | ほぼ対応だがモーダルの×が小さい | ナビが使えない/フォームが崩れる |
| V-2 | タブレットの独立体験 | スマホでもPCでもない最適レイアウト | 2列グリッド+適切なナビ切替 | PC表示をそのまま縮小 | スマホ表示が引き伸ばされる |
| V-3 | ギャラリーのタッチ体験 | スワイプで快適に写真を閲覧 | スワイプ+ピンチズーム+ライトボックス | スワイプのみ | タッチ操作不能/ホバー前提 |
| V-4 | フォーム送信の完走率 | スマホでフォーム入力→送信が快適 | 適切なtype+自動補完+バリデーション即表示 | type属性あるがバリデーション遅延 | テンキーが出ない/ボタンが画面外 |
| V-5 | 横スクロールゼロ | 全ページ全サイズで横スクロールなし | 375/768/1280px全てクリア | 1ページだけ微小な横スクロール | 複数ページで横スクロール発生 |
| V-6 | Tailwind体系の一貫性 | マジックナンバーなしでメンテナンス可能 | sm/md/lg/xl統一、カスタムBP=config定義 | 一部カスタムメディアクエリ | マジックナンバー乱立 |

## スコアリング

### 合計100点の内訳
| 層 | 配点 | 内容 |
|---|---|---|
| Layer 1: チェック検証可能性 | 30点 | 8項目全PASSで30点。1つでもFAILなら0点 |
| Layer 2: UX反映 | 40点 | 6項目、各項目の配点通り |
| Layer 3: 核の目的達成 | 30点 | 6項目、各5点 |

### 判定ルール
| スコア | 判定 | アクション |
|---|---|---|
| 90〜100 | **PASS** | そのまま次の機能へ。核が実現されている |
| 80〜89 | **CONDITIONAL** | Layer 2 or 3 の減点箇所を特定し修正 |
| 70〜79 | **FAIL** | UXか価値が不足。原因を記録し作り直し |
| 0〜69 | **CRITICAL FAIL** | チェック機能として動いていない。全体を作り直し |

### 合格ライン（チェック項目18項目ベース）
- **合格**: 18項目中 **16項目以上** クリア
  - **必須合格項目**（1つでも欠けたら不合格）: 横スクロールなし(1), タッチターゲット44px(2), ナビ動作(5), viewport(17)
- **条件付き合格**: **14〜15項目** クリア（修正指示付き納品可）
- **不合格**: **13項目以下**、または必須合格項目が1つでも欠けている場合

### この機能固有の重要判定ポイント
- **横スクロールは即死**: 1ページでも横スクロールが出たら不合格。原因の根本修正が必須
- **タッチターゲット44px**: フッターリンクやSNSアイコンが特に漏れやすい。padding込みで計測
- **テーブルのモバイル対応**: 料金表が崩れるのは顧客サイトで最も多い品質問題

## よくある不合格パターン

1. **固定幅の指定**
   - `w-[500px]` `w-[800px]` のようなピクセル固定幅
   - レスポンシブプレフィックスなしで使用
   - 375pxモバイルではみ出す

2. **画像のoverflow**
   - `next/image` に `fill` を使っているが親要素に `relative` と `overflow-hidden` がない
   - 画像がコンテナ外に飛び出す

3. **ハンバーガーメニューの未実装**
   - デスクトップのナビをそのままモバイルに表示
   - メニュー項目が折り返して潰れるか画面外に飛ぶ

4. **タッチターゲットの不足**
   - テキストリンクやアイコンボタンが32px以下
   - 特にフッターリンクやSNSアイコンで発生
   - ナビゲーション内の隣接するリンクが近すぎて誤タップ

5. **テーブルの溢れ**
   - 情報テーブル（会社概要・料金表・比較表等）がモバイルで横にはみ出す
   - ページ全体に横スクロールが発生する致命的パターン

6. **fixed要素の重複**
   - ヘッダー固定 + フローティングCTA + Cookie同意バーが同時表示
   - モバイルのコンテンツ表示領域が半分以下に

7. **グリッドのカラム未変更**
   - `grid-cols-3` や `grid-cols-4` のまま
   - モバイルでの `grid-cols-1` への切り替えを忘れ
   - カードが潰れて内容が読めない

8. **フォームのinput type未指定**
   - 電話番号欄に `type="tel"` がなくテンキーが出ない
   - メールアドレス欄に `type="email"` がなく `@` キーが出ない
   - URL欄に `type="url"` がない

9. **viewport未設定**
   - layout.tsxで `viewport` export を忘れ
   - モバイルでデスクトップ表示になる
   - ピンチズームしないと操作できない

10. **メニュー展開時の背景スクロール**
    - ハンバーガーメニュー展開時に背景が通常通りスクロール
    - メニューと背景が同時に動いて操作困難に
    - body の `overflow: hidden` 制御が漏れている

11. **モーダルの閉じるボタン画面外**
    - モバイルでモーダルが画面より大きくなる
    - 右上の×ボタンが見えない
    - 閉じる手段がなくなる致命的バグ

12. **カスタムメディアクエリの乱立**
    - Tailwindのブレークポイントを使わない
    - `@media (min-width: 847px)` のようなマジックナンバーが散在
    - チーム間で統一されずメンテナンス不能

13. **フォントサイズの非スケーリング**
    - デスクトップ用の `text-5xl`（48px）がモバイルでもそのまま
    - 1行に2〜3文字しか入らない
    - レスポンシブプレフィックス付きのサイズ指定を忘れている

14. **ヒーロー画像の高さ固定**
    - `h-[600px]` のような固定高さをヒーローに設定
    - モバイルでテキストとCTAが画面外にはみ出す
    - `min-h-screen` や `h-auto` でコンテンツに応じた高さにすべき

## 業種別の注意点

### 建築・建設業
- 施工実績の画像ギャラリーは写真が命
- モバイルでも高解像度でピンチズーム対応推奨
- Before/Afterの比較表示はモバイルでは縦並び（横並びだと各画像が小さすぎ）

### 飲食店
- メニュー表（料金テーブル）のモバイル対応が必須
- 横スクロールよりカード形式リフロー推奨
- 電話予約ボタンは `tel:` リンクでモバイルからワンタップ発信

### EC・小売
- 商品一覧のグリッドはモバイル2列が最適
  - 1列だとスクロール量過多、3列だと画像が小さすぎ
- カートへの追加ボタンは thumb zone（画面下半分）に配置
- フィルター/ソートのUIがモバイルで操作しやすいこと

### 医療・クリニック
- 予約フォームのモバイル対応は最重要
- 日付ピッカーがモバイルで操作しやすいこと
- 緊急連絡先の電話番号は常に画面内に表示

### 美容・サロン
- ギャラリー（施術例）はスワイプ操作でストレスなく閲覧
- ネット予約ボタンは全ページでモバイル画面内に到達可能な位置に
- スタッフ紹介のグリッドがモバイルで適切に縦積み

### 士業（弁護士・税理士等）
- 料金表のテーブルが複雑になりがち
- モバイルではアコーディオン形式への変換を推奨
- 無料相談の電話番号・フォームへの導線はモバイルで特に目立つ位置に

### ポートフォリオ・クリエイター
- 作品画像の表示品質がモバイルでも落ちないこと
- 圧縮しすぎに注意
- グリッドレイアウトはMasonry風でもモバイル1〜2列に正しくフォールバック

- **Google モバイルフレンドリーテスト**: モバイル対応の基準値（Lighthouse Mobile audit）

- **Tailwind CSS Responsive Design**: sm(640px), md(768px), lg(1024px), xl(1280px)
- **Chrome DevTools Device Mode**:
  - 375px (iPhone SE)
  - 390px (iPhone 14)
  - 768px (iPad)
  - 1280px (Desktop)
- **Lighthouse パフォーマンス監査**: モバイルでのレンダリングパフォーマンス（3Gシミュレーション）
- **Apple Human Interface Guidelines**: タッチターゲットサイズ推奨値 44pt
- **Material Design Guidelines**: タッチターゲットサイズ推奨値 48dp
- **Next.js Viewport Export**: App Routerの `viewport` export ドキュメント
