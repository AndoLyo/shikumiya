# しくみや — 全機能本番稼働 設計書

> 作成日: 2026-04-13
> ステータス: レビュー待ち

---

## 設計思想

この設計書は「機能の有無チェック」ではなく、**2つの視点**で全タスクを評価する。

1. **機能の核** — その機能が存在する本質的な理由。「何ができるか」ではなく「なぜ必要か」
2. **ユーザー視点** — ユーザーが実際に見て、触って、「これでいい」と判断する基準

スコアリングは各タスクに **必須項目（Pass/Fail）** と **品質スコア（0-100）** の2軸を設ける。
必須項目が1つでもFailなら、品質スコアに関わらず不合格。

---

## 全体アーキテクチャ（3つの流れ）

```
【流れ1】申込→サイト完成
  顧客が /start で申込 → Stripe決済 → webhook → サイト自動生成 → 顧客にURL通知

【流れ2】顧客がサイトを編集
  顧客が /member にログイン → 編集依頼を送信 → テキスト/画像は即反映、レイアウト/機能はClaude API経由

【流れ3】Lyoの運営管理
  Lyoが /admin で全顧客を一覧 → 依頼キュー処理 → MRR確認 → サブスク解約対応
```

## タスク 2-1: /member ダッシュボードの実データ接続

### 機能の核
> /member は「顧客が自分のサイトを管理する場所」。

- [ ] /member ログイン → `/api/member/auth` → GAS verify → 認証成功

- [ ] MemberCtx にリアルデータを注入（現在のハードコード「サンプル会社」を置換）

#### 2-1-B: ダッシュボード表示の実データ化

- [ ] ドメイン: GAS から取得した domain or siteUrl

- [ ] 編集回数: editsUsed / editsMax をリアルタイム表示

- [ ] 最近の依頼リスト: `/api/member/edit` の履歴を GAS から取得

#### 2-1-C: GAS 側の API 拡充
- [ ] `action=get_member_profile` — orderId + email で全プロフィール返却
- [ ] `action=get_edit_history` — orderId の編集依頼履歴を返却

## タスク 2-2: テキスト/画像の即時編集

### 機能の核
> 「写真を送るだけ。あとは全部おまかせ」が事業の核。
> 顧客が「このテキスト変えたい」「この写真変えたい」と思った瞬間に、選ぶだけで完了する。

### ユーザー視点
> 顧客は PC が苦手。「ここを変えたい」を選択肢から選ぶだけ。
> 変更がいつ反映されるかが見えること（「反映完了しました」の通知）。

### 作業内容

#### 2-2-A: edit-request ページの実データ化
- [ ] 現在の `CURRENT_SITE` ハードコードを、`/api/site-content` から取得に変更
- [ ] site.config.json の内容をセクション別に表示（hero, works, about, contact）
- [ ] 各セクションの現在値をフォームの初期値に設定

#### 2-2-B: テキスト変更 → /api/site-update 接続
- [ ] ユーザーがテキストフィールドを編集 → 送信 → `/api/site-update` を呼出

- [ ] 成功時: 「変更が反映されました」メッセージ + editsUsed のカウントアップ
- [ ] 失敗時: 具体的なエラーメッセージ（「月の編集回数を超えました」等）

- [ ] アップロード完了後 → `/api/site-update` で画像パス更新

## タスク 2-4: レイアウト/機能変更（Claude API 連携）

### 機能の核

**スコアリングの核**: Claude が生成したコードが「壊れていないか」「意図通りか」を自動判定する。
人間のレビュー負荷をゼロに近づけることが目的。

```
スコアリング 5軸（各20点、合計100点）:

1. ビルド成功（20点）

   - 0点: ビルドエラー → 即却下
   - 20点: エラーゼロ

   - TypeScript の型エラーがないか
   - site-config-schema.ts の型と整合するか
   - 0点: 型エラーあり
   - 20点: 型エラーゼロ

3. 意図一致（20点）
   - 変更依頼の内容と実際の差分が一致するか
   - Claude API のセルフチェック（変更内容を要約させて依頼と比較）
   - 0-20点: 一致度に応じて

4. 既存機能の保全（20点）
   - 変更前に存在したセクションが消えていないか
   - DOM の構造比較（変更前後のセクション数）
   - 0点: セクションが消えた

   - インデントの一貫性
   - 0-20点: 品質に応じて
```
